# BingWall 模块说明

## 文档元信息

- 更新时间：2026-03-23T12:57:44Z
- 依据文档：`docs/system-design.md`
- 文档定位：实施前模块划分、职责边界和建议目录说明

## 目标

本文件用于把系统设计中的五层架构落实为后续开发可执行的模块边界，避免在代码实现阶段出现职责混乱、接口耦合或目录失控。

## 模块总览

一期建议按以下模块组织：

| 模块 | 核心职责 | 主要输入 | 主要输出 | 不负责内容 |
|---|---|---|---|---|
| 采集模块 | 获取 Bing 数据、规范化、去重、下载资源、落库 | 调度触发信号、手动采集任务、上游数据 | 壁纸记录、资源记录、任务记录、日志 | 页面渲染、公开查询 |
| 领域模型模块 | 封装内容、资源、任务、状态流转规则 | 采集结果、后台操作、数据库记录 | 统一业务对象和校验逻辑 | HTTP 路由、定时调度 |
| 存储模块 | 读写 SQLite、本地文件和操作日志 | 业务对象、查询条件、文件流 | 持久化记录、资源路径、审计记录 | 页面展示、任务调度 |
| 公开 API 模块 | 提供列表、详情、筛选、站点信息接口 | 访客请求、查询参数 | 统一响应结构 | 后台鉴权、采集执行 |
| 后台 API 模块 | 登录、内容管理、任务管理、日志查看 | 管理员请求、鉴权信息、操作命令 | 管理操作结果、审计日志 | 定时调度、页面渲染 |
| 前端展示模块 | 渲染公开站点页面 | 公开 API、图片静态地址 | 首页、列表页、详情页 | 数据修改、后台功能 |
| 管理后台模块 | 渲染后台页面并调用后台 API | 后台 API、管理员操作 | 管理视图、任务视图、日志视图 | 底层文件系统操作 |
| 调度与运维模块 | 托管进程、触发任务、备份巡检、健康检查 | systemd、cron、运维脚本 | 触发信号、备份文件、巡检结果、健康状态 | 业务页面与接口逻辑 |

## 模块边界说明

### 1. 采集模块

负责内容：

- 接收调度模块发出的自动采集或任务消费触发信号
- 拉取 Bing 上游元数据
- 将上游字段转换为统一内部结构
- 按业务主键和技术辅助键去重
- 下载图片到临时目录并完成校验
- 写入壁纸主体、图片资源、采集任务和明细日志

不负责内容：

- 不直接对外暴露 HTTP 接口
- 不直接控制公开展示页面
- 不承担后台页面逻辑

### 2. 领域模型模块

负责内容：

- 统一内容状态和资源状态规则
- 对业务对象做运行时校验
- 定义状态流转限制
- 提供面向服务层的业务操作

不负责内容：

- 不直接做数据库连接管理
- 不负责调度与部署

### 3. 存储模块

负责内容：

- 封装 SQLite 的读写逻辑
- 封装图片临时目录、正式目录和失败隔离目录
- 提供资源定位和文件存在性检查
- 保存任务日志和审计日志

不负责内容：

- 不承担业务决策
- 不暴露原始磁盘路径给前端或后台页面

### 4. 公开 API 模块

负责内容：

- 参数校验
- 统一响应格式
- 壁纸列表、详情、筛选、站点信息接口
- 仅查询公开可见的数据

不负责内容：

- 不执行采集
- 不修改内容状态

### 5. 后台 API 模块

负责内容：

- 管理员登录与会话管理
- 内容启用、禁用、逻辑删除
- 手动采集任务创建和查询
- 日志与审计查询

不负责内容：

- 不在请求链路中执行长时间采集下载任务
- 不暴露数据库调试接口或任意文件浏览能力

### 6. 前端展示模块

负责内容：

- 首页、列表页、详情页展示
- 用户筛选、分页、下载入口
- 空状态与错误页展示

不负责内容：

- 不管理权限
- 不维护内容状态

### 7. 管理后台模块

负责内容：

- 登录页、内容管理页、任务页、日志页
- 展示后台 API 返回的管理数据
- 对危险操作做二次确认

不负责内容：

- 不直接改数据库文件
- 不直接访问图片目录

### 8. 调度与运维模块

负责内容：

- `systemd` 托管应用进程
- `cron` 触发自动采集、手动任务消费、巡检、备份
- 健康检查与日志归档
- 执行数据库与文件系统层面的运维脚本

不负责内容：

- 不改写业务规则
- 不直接提供页面功能
- 不定义“内容是否可公开”“资源是否应下线”等业务判定规则

补充说明：

- 调度与运维模块只负责按时间触发脚本或接口，不直接承载采集去重、状态流转和公开规则判断
- 深度健康检查中的“最近一次采集状态”等业务摘要，应由 API 层或领域服务读取任务记录后生成，运维模块只负责触发检查和收集结果

## 模块依赖原则

- API 模块依赖领域模型模块和存储模块，但不直接依赖调度实现
- 采集模块依赖领域模型模块和存储模块
- 前端模块只依赖 API 契约和静态资源地址
- 调度与运维模块只触发服务或脚本，不写业务逻辑
- 所有模块都必须遵循统一配置入口，环境相关信息不得写死

## 实施约束

- 一期不做微服务拆分
- 一期不引入消息队列
- 一期不新增与设计总纲冲突的服务模块
- 如后续需要拆分模块，必须先更新系统设计和本文件
