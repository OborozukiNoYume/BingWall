# BingWall 搜索与查询性能基线报告

## 文档元信息

- 更新时间：2026-04-05T05:47:44Z
- 基准脚本：`scripts/benchmark_public_queries.py`
- 基准目标：为公开列表与后台内容列表的关键词、标签、日期类查询建立可重复的 SQLite 基线，并给出额外索引 / FTS 的升级阈值
- 执行环境：本仓库本地 `uv run python scripts/benchmark_public_queries.py`，运行时输出显示 `Python 3.14.3`、`SQLite 3.50.4`

## 适用边界

- 本报告覆盖公开列表 `/api/public/wallpapers` 与后台内容列表 `/api/admin/wallpapers` 对应的仓储 / 服务层查询路径
- 公开列表重点覆盖 `keyword`、`tag_keys`、`date_from`、`date_to` 四类过滤组合
- 后台内容列表重点覆盖 `keyword`、`content_status`、`market_code`、`created_from_utc`、`created_to_utc` 组合
- 当前公开列表里的 `market_code` 主要用于本地化内容选择，不构成独立 SQL 过滤条件；因此本轮未把它单列为公开 SQL 基准项

## 复跑命令

```bash
uv run python scripts/benchmark_public_queries.py
```

默认口径：

- 自动创建临时 SQLite 数据库并执行现有迁移
- 默认造数 `12,000` 条壁纸、`24,000` 条本地化、`24,000` 条图片资源、`36,000` 条内容标签绑定
- 每个场景默认执行 `5` 次预热和 `30` 次正式测量
- 输出 `P50`、`P95`、`P99`、最大值和当前升级建议

| 指标 | 数值 |

说明：

- 样本中同时包含 `enabled + public`、`enabled + non-public`、`disabled`、`draft` 内容，避免只用全公开数据导致后台列表基线失真
- 关键词命中路径同时覆盖主表文本、本地化文本和标签文本，能反映当前 `LIKE + EXISTS` 查询形态
- 日期范围样本按 `8` 个市场平铺生成，便于覆盖当前项目的典型 Bing 多市场分布

## 升级阈值

当前建议按以下规则判断是否需要继续加索引或引入 FTS：

| 触发项 | 处理建议 |
| --- | --- |
| 纯日期 / 时间过滤场景的 `P95` 超过本文阈值，或相对本基线退化超过 `2x` | 先检查现有排序和过滤字段是否缺少常规 B-Tree 索引 |
| 任一关键词敏感场景达到 `REVIEW`，即 `P95` 或 `P99` 超过该场景阈值 | 进入 `FTS` 评估，优先比较 `SQLite FTS5` 与当前 `LIKE + EXISTS` 的收益 |
| `wallpapers` 规模增长到约 `50,000` 行以上，且关键词检索已成为高频入口 | 即使阈值尚未超线，也应重新跑基准并评估 `FTS` |
| 新增会显著放大搜索面的字段，例如更多本地化文本列、更多标签层级或更复杂排序 | 先补基准，再决定是否改索引或引入全文检索 |

## 当前结论

- 在默认 `12,000` 条壁纸样本下，当前 `SQLite + LIKE + EXISTS` 方案仍有明显余量
- 日期类过滤基本在 `1ms` 左右完成，现有 `wallpaper_date` / `created_at_utc` 索引口径足够
- 关键词相关场景 `P95` 均低于 `26ms`，离本轮升级阈值还有较大空间
- 因此本阶段结论是：继续保持当前实现，不新增额外索引，不引入 `FTS`
