# BingWall 图片服务系统设计说明书

**文档版本**：V1.0  
**文档日期**：2026-03-23  
**文档性质**：系统设计文档  
**适用阶段**：一期单机落地实施

## 1. 项目概述

BingWall 是一个围绕 Bing 壁纸构建的图片服务系统。系统不以“下载脚本”为目标，而是以“可持续采集、可管理、可对外服务、可扩展演进”的内容系统为目标，形成从数据采集、资源存储、接口服务、公开展示到后台管理的完整闭环。

系统一期采用单机架构，围绕以下五个核心部分建设：

1. 采集系统：自动或手动获取 Bing 壁纸及其元数据。
2. 存储层：持久化保存内容数据与图片文件。
3. API 服务层：向公开前端和管理后台提供业务接口。
4. 公开前端：为访客提供浏览、筛选、详情和下载能力。
5. 管理后台：为管理员提供内容管理、状态控制、手动采集和日志查看能力。

本方案面向工程落地，重点解决以下问题：

- 如何稳定采集 Bing 壁纸并形成可管理数据。
- 如何在 SQLite、本地文件、cron、单机部署条件下形成可运行闭环。
- 如何为后续扩展到多来源、搜索、标签、OSS 和统计分析保留演进空间。
- 如何在不引入复杂分布式基础设施的前提下保证可维护性和可运营性。

## 2. 项目目标与范围

### 2.1 项目目标

本项目的目标是建设一个正式可运行的图片服务系统，满足以下要求：

- 自动定时采集 Bing 壁纸。
- 支持后台手动触发采集。
- 保存壁纸图片和业务元数据。
- 对外提供公开 API。
- 提供公开浏览与下载页面。
- 提供后台管理能力。
- 为后续扩展多来源、搜索、标签、OSS 和统计能力保留接口和数据空间。

### 2.2 一期范围

一期明确纳入范围的能力如下：

| 范围项 | 说明 |
|---|---|
| Bing 数据采集 | 采集壁纸元数据与图片文件 |
| 内容持久化 | 保存壁纸、资源、任务、日志、后台账号等数据 |
| 公开 API | 提供壁纸列表、详情、筛选、下载相关接口 |
| 管理 API | 提供登录、内容管理、手动采集、日志查看接口 |
| 公开前端 | 无登录浏览、筛选、查看详情、下载 |
| 管理后台 | 登录后管理内容状态、执行采集、查看日志 |
| 单机部署 | Ubuntu + Nginx + FastAPI + systemd + cron |
| 本地文件存储 | 开发与一期生产均可使用本地目录 |

### 2.3 明确不纳入一期范围

以下内容不属于一期交付范围：

- 公开用户注册、登录、个人中心
- 微服务拆分
- 分布式任务调度
- 复杂全文检索系统
- 对象存储作为唯一存储方案
- 多租户能力
- 智能推荐、个性化订阅
- 复杂数据分析平台

### 2.4 范围控制原则

一期方案坚持最小可用闭环，不引入与当前业务规模不匹配的复杂基础设施。所有设计优先服务于“稳定采集、可管理、可展示、可下载”四个核心目标。

## 3. 用户角色与使用场景

### 3.1 用户角色

本系统至少包含以下角色：

| 角色 | 主要目标 | 是否登录 |
|---|---|---|
| 访客 | 浏览、筛选、查看详情、下载壁纸 | 否 |
| 管理员 | 管理内容、控制状态、手动采集、查看日志 | 是 |
| 运维人员 | 部署、巡检、备份、恢复、日志排查 | 是，系统级 |

### 3.2 核心使用场景

#### 访客场景

- 打开首页查看最新壁纸。
- 按日期、地区、分辨率筛选图片。
- 进入详情页查看标题、说明、来源、发布日期等信息。
- 下载图片原图或指定资源版本。

#### 管理员场景

- 登录后台查看采集结果。
- 手动发起一次采集任务。
- 对内容进行启用、禁用、删除。
- 查看失败任务和异常日志。
- 核查重复数据或图片处理失败情况。

#### 运维场景

- 确认服务健康状态。
- 查看采集成功率和接口状态。
- 执行数据库与图片文件备份。
- 在异常场景下恢复数据库和资源目录。

## 4. 技术选型与可行性分析

### 4.1 技术选型

| 层面 | 选型 |
|---|---|
| 后端 | Python + FastAPI |
| 数据库 | SQLite |
| 定时任务 | cron |
| Web 服务 | Nginx |
| 进程托管 | systemd |
| 图片存储 | 本地目录 |
| 部署环境 | Ubuntu |
| 前端 | 公开前端 + 管理后台，均通过 API 访问后端 |

### 4.2 选型理由

#### Python + FastAPI

适合作为一期后端基础，原因如下：

- 开发效率高，适合中小型内容服务系统。
- 天然适合接口服务、后台管理、采集逻辑整合。
- 生态成熟，便于后续接入 ORM、迁移工具、认证组件、对象存储 SDK 等能力。
- 对单机部署友好，运维复杂度低。

#### SQLite 在当前阶段可行

SQLite 在一期是可行方案，原因如下：

- 采集频率低，Bing 壁纸新增节奏通常为每日级或少量批量级。
- 公开站点在一期访问量有限，读多写少。
- 数据模型相对简单，以内容、资源、任务、日志为主。
- 单机部署下，SQLite 部署简单、备份方便、恢复成本低。
- 无需独立数据库服务，减少运维负担。

SQLite 的明确技术边界如下：

- 不适合高并发写入。
- 不适合多个应用节点同时写。
- 不适合复杂统计和大规模全文检索。

这些限制在一期是可接受的技术债，因为本项目当前不追求高并发写入与分布式扩展。

#### 本地文件存储在当前阶段可行

本地文件存储适合作为一期方案，原因如下：

- 图片量初期有限，文件规模可控。
- 与单机部署天然匹配，路径管理清晰。
- 静态文件可由 Nginx 高效提供访问。
- 备份策略简单，数据库与资源目录可同时打包。

其边界和技术债如下：

- 无跨节点共享能力。
- 无内建高可用。
- 扩容时需要迁移到共享存储或对象存储。

这些问题在一期接受，因为当前架构是单机，目标是尽快完成业务闭环，而不是先行搭建复杂存储体系。

#### cron 在当前阶段可行

cron 适合一期定时采集，原因如下：

- 任务类型单一，主要是定时拉取 Bing 数据。
- 调度频率低，对秒级调度无要求。
- 与 Ubuntu 环境兼容性好，运维成本低。
- 可以明确区分“定时触发”和“业务执行”，排障路径简单。

其局限性如下：

- 缺乏复杂任务编排能力。
- 缺少任务可视化能力。
- 不适合大量并发任务和复杂依赖链。

一期接受这些限制，因为系统只需处理少量固定周期任务，复杂调度能力暂不构成刚性需求。

#### 单机架构是合理的

单机架构满足当前阶段目标，原因如下：

- 系统角色集中，模块间调用简单。
- 采集、存储、接口、前端管理均可在单节点闭环。
- 业务规模尚不足以支撑多机架构带来的复杂性成本。
- 有利于快速迭代、快速上线、快速验证。

明确接受的技术债包括：

- 单点故障。
- 水平扩展能力不足。
- 本地磁盘容量受限。
- 后台任务能力有限。

这些技术债并非忽略，而是在一期范围内有意识接受，并在后续扩展章节给出升级路径。

## 5. 系统总体架构

### 5.1 架构目标

总体架构以“分层清晰、职责可控、易于演进”为目标，采用单机分层架构，不进行微服务拆分。

### 5.2 总体架构说明

系统逻辑上分为五层：

1. 采集系统
2. 存储层
3. API 服务层
4. 公开前端
5. 管理后台

其关系如下：

- 采集系统从 Bing 获取数据，经统一规范化后写入存储层。
- API 服务层从存储层读取公开数据，为前端提供查询能力；同时向后台提供管理能力。
- API 路由按职责分为 `/api/public`、`/api/admin` 和 `/api/health` 三类前缀。
- 公开前端只依赖公开 API 和静态图片地址。
- 管理后台只依赖后台 API，不直接操作数据库和文件系统。
- Nginx 负责反向代理 API、托管前端静态资源和图片资源。
- systemd 负责托管 FastAPI 服务进程。
- cron 负责触发定时采集任务。

### 5.3 架构分层原则

| 分层 | 核心职责 | 不负责内容 |
|---|---|---|
| 采集系统 | 获取上游数据、标准化、去重、落库、下载图片 | 对外提供公开展示 |
| 存储层 | 保存结构化数据与图片文件 | 业务规则编排、页面展示 |
| API 服务层 | 参数校验、业务编排、鉴权、状态控制、数据输出 | 定时调度、磁盘巡检 |
| 公开前端 | 页面展示、筛选交互、下载引导 | 内容修改、后台管理 |
| 管理后台 | 内容管理、任务触发、日志查看 | 定时调度执行、底层文件写入 |

## 6. 模块划分与职责边界

### 6.1 采集系统

#### 负责内容

- 接收定时任务或手动采集请求。
- 调用 Bing 上游接口或页面获取壁纸元数据。
- 将上游数据转换为统一内部结构。
- 执行去重判断。
- 下载图片文件。
- 记录采集任务结果与失败原因。
- 更新内容状态和图片处理状态。

#### 不负责内容

- 不负责对外提供页面展示。
- 不负责前端交互逻辑。
- 不负责直接提供公共下载页面。
- 不负责业务审核策略以外的后台展示。

#### 输入

- cron 定时触发信号。
- 管理后台发起的手动采集任务。
- 源配置，例如地区、日期范围、采集模式。

#### 输出

- 标准化后的壁纸元数据记录。
- 本地图片文件。
- 资源记录。
- 任务执行日志。
- 成功、失败、重复等处理结果。

### 6.2 存储层

#### 负责内容

- 保存壁纸主体数据。
- 保存图片资源记录。
- 保存采集任务与日志。
- 保存后台账号和操作审计。
- 为 API 提供可查询数据基础。

#### 不负责内容

- 不负责业务调度。
- 不负责页面渲染。
- 不负责认证流程控制。
- 不负责文件下载鉴权逻辑。

#### 输入

- 采集系统写入的结构化数据和图片文件。
- 管理后台触发的状态变更和操作日志。
- API 服务的查询请求。

#### 输出

- 可读取的结构化记录。
- 可访问的资源路径。
- 可追踪的任务与审计信息。

### 6.3 API 服务层

#### 负责内容

- 对外提供公开查询接口。
- 对后台提供登录、管理、任务、日志接口。
- 统一参数校验、错误处理和响应规范。
- 执行业务规则，如是否公开展示、是否允许下载。
- 输出页面需要的数据结构。
- 生成请求追踪标识并记录日志。

#### 不负责内容

- 不负责定时调度。
- 不负责页面静态资源托管。
- 不负责复杂离线任务处理。

#### 输入

- 公开前端请求。
- 管理后台请求。
- 鉴权信息、查询参数、状态变更命令。

#### 输出

- 统一业务响应。
- 公开列表与详情数据。
- 管理操作结果。
- 错误码和审计日志。

### 6.4 公开前端

#### 负责内容

- 浏览最新壁纸。
- 按条件筛选和分页。
- 查看详情。
- 发起下载。
- 展示错误页和空结果提示。

#### 不负责内容

- 不负责修改内容状态。
- 不负责执行采集。
- 不负责直接访问数据库。
- 不负责管理操作。

#### 输入

- 公开 API 返回的数据。
- 图片静态资源地址。

#### 输出

- 用户可见页面。
- 筛选条件和下载动作。

### 6.5 管理后台

#### 负责内容

- 管理员登录。
- 图片内容管理。
- 状态控制。
- 手动采集入口。
- 任务结果查看。
- 日志查看和问题定位辅助。

#### 不负责内容

- 不负责执行底层采集逻辑。
- 不负责直接修改数据库文件。
- 不负责公开站点渲染。

#### 输入

- 后台 API。
- 管理员操作命令。

#### 输出

- 管理视图。
- 操作结果反馈。
- 审计痕迹。

## 7. 核心业务流程

### 7.1 自动采集流程

1. cron 按预设周期触发采集命令。
2. 采集系统创建一条任务执行记录，状态设为运行中。
3. 系统向 Bing 上游发起元数据请求。
4. 上游返回数据后，系统进行字段映射和规范化。
5. 对每条候选内容执行去重判断。
6. 若为新内容，则先写入壁纸主体记录，内容状态初始为 `draft`，图片处理状态默认为 `pending`。
7. 采集系统下载图片到本地临时目录。
8. 下载成功后进行文件校验，生成资源记录并转入正式目录。
9. 图片处理状态更新为 `ready`。
10. 当前默认采用自动发布策略：当资源全部就绪后，内容状态从 `draft` 转为 `enabled`，并设置为允许公开展示；若关闭自动发布配置，则保持 `draft`。
11. 任务完成后记录成功数、重复数、失败数、耗时和错误摘要。
12. 若整个任务失败，则任务记录标记失败并保留失败原因。

### 流程目标

- 保证自动采集可重复执行。
- 重复执行不会产生脏数据。
- 任一条目失败不应导致全部已成功条目回滚。

### 7.2 手动采集流程

一期手动采集采用“后台提交任务、cron 近实时消费”的保守方案，不在接口请求链路中执行长时间下载任务。

1. 管理员在后台发起手动采集。
2. 后台 API 校验参数，例如地区、日期、是否强制刷新。
3. 系统写入一条手动采集任务记录，状态为待执行。
4. cron 每分钟扫描待执行任务并开始处理。
5. 后续处理流程与自动采集完全一致。
6. 管理后台轮询或刷新任务状态，查看执行结果。
7. 若失败，后台显示失败原因，管理员可再次重试。

### 选择该方案的原因

- 不阻塞后台接口请求。
- 不需要引入任务队列。
- 自动采集与手动采集复用同一条业务路径，降低分叉风险。

### 7.3 公开前端访问流程

1. 访客进入首页。
2. 前端调用公开列表接口获取默认数据集。
3. API 服务按公开规则查询 `enabled` 且图片状态为 `ready` 的内容。
4. 前端渲染图片卡片、基础元数据和筛选项。
5. 用户点击某一项进入详情页。
6. 前端请求详情接口，获取完整展示信息和下载入口。
7. 下载动作直接访问静态资源地址，或由下载接口完成跳转与统计。
8. 若列表为空、详情不存在、资源不可用，则前端展示明确错误或空状态页面。

### 7.4 后台管理流程

1. 管理员登录后台。
2. 后台调用管理 API 获取内容列表、任务记录和异常摘要。
3. 管理员可执行启用、禁用、逻辑删除、重试采集、查看日志等操作。
4. API 服务校验管理员权限后更新数据库，并记录操作审计。
5. 被禁用或删除的内容不再出现在公开前端。
6. 管理员可进入详情页查看来源字段、资源状态、失败原因和最近操作记录。

### 7.5 异常流程

异常流程应遵循“可定位、可恢复、不扩大损害”的原则。

- 若上游网络失败，任务记录失败并保留可重试状态，不修改已有成功内容。
- 若单张图片下载失败，仅该条记录进入 `failed`，不影响其他条目。
- 若写库失败，任务记录异常终止，并保留临时文件清理记录。
- 若发现重复数据，记为重复跳过，不视为系统错误。
- 若上游格式变化导致解析失败，记录原始摘要并触发告警，避免错误数据进入业务表。

## 8. 数据模型设计

### 8.1 设计原则

数据模型必须服务业务，而不是仅仅保存字段。字段设计应覆盖以下五类目的：

1. 展示用途：给公开前端和详情页使用。
2. 状态控制：决定内容是否可见、资源是否可用。
3. 资源管理：定位文件、核验资源、支持迁移。
4. 任务追踪：定位采集过程中的成功、失败和耗时。
5. 扩展预留：支持多来源、标签、搜索、OSS、统计。

### 8.2 核心实体

一期建议至少包含以下实体：

| 实体 | 作用 |
|---|---|
| 壁纸主体 | 表示一条可管理的壁纸内容 |
| 图片资源 | 表示与壁纸关联的实际图片文件 |
| 采集任务 | 表示一次自动或手动采集执行 |
| 采集明细日志 | 表示任务中的逐条处理结果 |
| 管理员账号 | 后台登录使用 |
| 操作审计日志 | 记录后台管理动作 |

### 8.3 壁纸主体实体

壁纸主体是业务核心，表示“这张壁纸作为一条内容”的管理单元。

#### 字段分组与业务意义

##### 标识字段

- 主键 ID：系统内部唯一标识。
- 来源类型：用于区分当前是 Bing，后续可扩展其他来源。
- 来源唯一键：用于跨来源去重和追踪。
- 地区代码：用于区分不同市场版本。
- 壁纸日期：用于按日期组织和展示。

这些字段用于唯一识别和去重，不直接面向用户展示，但决定内容归属。

##### 展示字段

- 标题
- 副标题或简述
- 版权说明
- 来源名称
- 发布时间
- 拍摄地或主题说明
- 详情描述
- 预览图信息

这些字段直接服务列表页和详情页，是用户可见内容。

##### 状态控制字段

- 内容状态
- 是否允许公开展示
- 是否允许下载
- 发布开始时间
- 发布结束时间
- 删除标记
- 创建时间
- 更新时间

这些字段决定内容是否可被前端查询到，属于运营控制核心字段。

##### 资源关联字段

- 默认资源 ID
- 原始来源 URL
- 原始页面 URL
- 原始图片尺寸信息
- 当前可用资源状态

这些字段用于内容与图片资源之间的关系管理，决定下载和详情页展示资源是否可用。

##### 扩展字段

- 标签预留字段或关联关系
- 搜索关键词预留字段
- 来源原始扩展信息
- 多语言标题预留空间
- 排序权重

这些字段为后续多来源、标签搜索、运营排序和国际化留出空间。

### 8.4 图片资源实体

图片资源表示实际文件，不应与壁纸主体混为一体。原因是未来一条内容可能对应多个资源版本，例如原图、缩略图、不同分辨率版本、OSS 镜像地址。

#### 字段分组与业务意义

##### 标识与归属

- 资源 ID
- 壁纸 ID
- 资源类型，例如原图、缩略图、下载图

##### 文件管理字段

- 存储后端类型，例如本地、OSS
- 存储相对路径
- 文件名
- 文件扩展名
- MIME 类型
- 文件大小
- 宽度
- 高度

这些字段用于定位和输出资源，不仅服务下载，也为迁移和展示提供依据。

##### 校验字段

- 源 URL
- URL 哈希
- 文件内容哈希
- 下载时间
- 校验结果

这些字段用于去重、完整性校验和后续资源治理。

##### 状态字段

- 图片处理状态
- 失败原因
- 最近处理时间

这些字段决定资源是否真正可用，是公开下载前的门槛。

### 8.5 采集任务实体

采集任务用于记录一次完整执行，必须独立建模，不能只依赖日志文件。

#### 关键意义

- 区分自动任务和手动任务。
- 支持后台查看执行结果。
- 支持失败重试和问题追踪。
- 支持未来扩展到更多来源。

#### 核心字段

- 任务 ID
- 任务类型
- 来源类型
- 触发方式
- 触发人
- 任务状态
- 开始时间
- 结束时间
- 成功数量
- 重复数量
- 失败数量
- 错误摘要
- 请求参数快照

### 8.6 采集明细日志实体

该实体用于记录任务中每个候选条目的处理结果，解决“任务失败但不知道失败在哪”的问题。

建议保存以下信息：

- 所属任务 ID
- 候选来源标识
- 处理动作
- 处理结果
- 失败原因
- 去重命中类型
- 文件写入结果
- 数据库写入结果
- 时间戳

### 8.7 管理员账号与审计日志

虽然项目不要求公开用户系统，但后台必须具备基本管理员能力。

管理员账号需要：

- 用户名
- 密码摘要
- 角色或权限级别
- 状态
- 最近登录时间

操作审计日志需要：

- 操作人
- 操作类型
- 目标对象
- 操作前后状态摘要
- 请求来源
- 时间戳
- 追踪 ID

## 9. 状态模型与数据生命周期

### 9.1 内容状态

内容状态用于控制“这条壁纸作为内容是否对外可见”。

| 状态 | 含义 | 是否公开可见 |
|---|---|---|
| `draft` | 已入库但未发布，通常用于待确认或资源未就绪 | 否 |
| `enabled` | 已启用，可在公开前端展示 | 是 |
| `disabled` | 已下线，内容保留但不再公开显示 | 否 |
| `deleted` | 逻辑删除，不再作为有效内容参与业务 | 否 |

#### 状态流转原则

- 新采集内容会先进入 `draft`。
- 当前默认配置下，当图片资源可用且满足发布策略时，会自动转为 `enabled`；若关闭自动发布配置，则继续保持 `draft` 等待后台确认。
- 已启用内容可由管理员转为 `disabled`。
- `disabled` 可重新启用回到 `enabled`。
- 任意非删除状态可逻辑删除为 `deleted`。
- `deleted` 不建议直接恢复为 `enabled`，应先恢复到 `disabled` 或 `draft`，再由管理员确认发布。

### 9.2 图片处理状态

图片处理状态用于控制“资源文件是否真实可用”。

| 状态 | 含义 |
|---|---|
| `pending` | 已创建资源记录但尚未完成下载、校验或入库 |
| `ready` | 图片文件可用，资源信息完整，可用于展示或下载 |
| `failed` | 下载、校验、写入、移动等任一步失败 |

#### 状态流转原则

- 新建资源记录默认 `pending`。
- 下载并校验成功后转为 `ready`。
- 下载失败、文件损坏、写入失败则转为 `failed`。
- `failed` 允许通过重试重新进入 `pending`，重试成功后转为 `ready`。

### 9.3 状态组合约束

内容状态与图片处理状态必须联动约束：

- 内容不能在图片状态为 `pending` 或 `failed` 时进入公开展示。
- 公开接口只返回同时满足以下条件的数据：内容状态为 `enabled`、图片状态为 `ready`、允许公开展示，且当前时间处于发布时间窗口内（如设置了开始或结束时间）。
- 详情页可展示公开内容本身，但下载入口仅在允许下载时返回；若不允许下载，应返回不可下载状态或不返回下载地址。
- 若已启用内容对应资源后续损坏，则应自动降级为 `disabled` 或从公开查询中排除。
- `deleted` 内容不要求立即删除文件，但必须从公开查询中彻底隔离。

### 9.4 数据生命周期

1. 候选数据被采集。
2. 通过去重后生成 `draft` 内容。
3. 资源进入 `pending`。
4. 图片准备完成后资源变为 `ready`。
5. 内容根据策略转为 `enabled`。
6. 若运营下线则转为 `disabled`。
7. 若逻辑删除则转为 `deleted`。
8. 到达清理条件后，可对文件执行延迟物理清理。

## 10. 去重策略

### 10.1 去重目标

去重策略必须解决两个问题：

- 避免同一壁纸被重复入库。
- 保留未来多来源、多版本、多地区下的可扩展性。

### 10.2 方案比较

| 方案 | 优点 | 缺点 | 适用阶段 |
|---|---|---|---|
| 壁纸日期 + 地区 | 业务含义强，适合 Bing 每日壁纸模式 | 无法覆盖同日多图、改图重发、跨来源情况 | 一期推荐主策略 |
| `source_url` | 实现简单，便于技术去重 | 上游 URL 可能变化、失效或带临时参数 | 一期推荐辅策略 |
| URL hash | 便于存储和索引，适合长 URL | 本质上仍依赖 URL 稳定性 | 一期可作为技术字段 |
| 图片内容 hash | 去重最准确，可跨 URL、跨来源识别相同内容 | 需要下载文件后计算，成本更高 | 二期推荐增强策略 |

### 10.3 一期推荐方案

一期推荐采用“双层去重”：

1. 业务主键去重：`来源类型 + 壁纸日期 + 地区代码`
2. 技术辅助去重：`source_url` 或其哈希值

#### 推荐原因

- Bing 的业务模式与日期、地区高度相关，适合作为主去重依据。
- 仅依赖 URL 不够稳健，因为上游可能调整图片参数或链接结构。
- 仅依赖内容哈希会把去重放到下载后，无法在下载前拦截明显重复。
- 双层策略可以先低成本拦截大部分重复，再保留后续通过内容哈希做精确治理的能力。

### 10.4 去重处理原则

- 命中业务主键且资源正常时，视为重复，跳过新增。
- 命中业务主键但资源状态为 `failed` 时，可按重试策略进行资源修复，不应重复创建内容。
- 命中 URL 去重但业务主键不同，应记录为可疑重复，留待后台核查。
- 内容哈希在后续引入时，应作为“资源级去重”和“跨来源同图识别”的增强策略，而不是替代一期业务主键。

## 11. 图片资源策略

### 11.1 存储策略

一期采用本地目录存储，建议按来源、年份、月份、地区进行目录分层，避免单目录文件过多，便于归档、备份和迁移。

建议管理三类路径：

- 临时下载目录
- 正式资源目录
- 失败隔离目录或日志定位目录

#### 原因

- 临时目录与正式目录分离，避免半成品文件被公开访问。
- 正式目录结构化，便于迁移到 OSS 时保持路径语义一致。
- 失败文件独立处理，有利于问题定位和清理。

### 11.2 文件命名原则

文件命名应满足以下原则：

- 可读但不依赖标题文本。
- 优先使用稳定业务键组合。
- 避免中文或特殊字符作为文件名核心。
- 保证重试下载时命名稳定。
- 同名冲突可追加短哈希。

### 11.3 资源版本策略

一期至少支持原图资源。缩略图、WebP、不同分辨率版本属于后续扩展能力，但资源模型需要预留能力。

建议资源类型至少预留：

- 原图
- 列表缩略图
- 详情预览图
- 下载图

即使一期只落地原图，也应从数据模型上区分“内容”和“资源”。

### 11.4 图片访问策略

图片访问有两种方式：

#### 静态 URL 访问

由 Nginx 直接提供本地文件访问。

优点：

- 性能高，链路短。
- FastAPI 不参与大文件传输，减轻应用压力。
- 配置简单，适合单机。

缺点：

- 下载行为统计能力较弱。
- 精细权限控制能力弱。
- 防盗链和限速控制需由 Nginx 完成。

#### 代理下载

由 API 服务接管下载请求，再读取文件或重定向。

优点：

- 便于记录下载日志。
- 便于做权限、签名、频率控制。
- 便于未来接 OSS、CDN 或鉴权下载。

缺点：

- 增加应用层开销。
- 实现复杂度更高。
- 大文件下载会占用应用资源。

#### 一期建议

一期建议以静态 URL 为主，公开前端下载直接访问静态资源，由 Nginx 提供文件服务。  
如需记录下载量，可增加轻量级“下载登记接口 + 静态资源跳转”的折中方案，但不应让 FastAPI 成为大文件传输主链路。

## 12. API 设计原则

### 12.1 API 分类边界

#### 公开接口范围

公开接口仅服务访客浏览和下载，不包含任何管理能力。

应包括：

- 壁纸列表查询
- 壁纸详情查询
- 筛选条件查询
- 下载入口相关接口或地址信息
- 站点基础信息接口

不应包括：

- 手动采集
- 内容状态修改
- 日志查询
- 后台账号管理

#### 后台接口范围

后台接口仅服务管理员，必须有鉴权控制。

应包括：

- 登录与会话管理
- 内容管理
- 状态切换
- 手动采集任务创建
- 任务列表与详情
- 失败重试
- 日志查看
- 审计记录查询

不应包括：

- 无约束的文件系统浏览
- 原始数据库调试接口
- 无审计的批量危险操作

### 12.2 统一响应结构原则

除静态文件和特殊下载响应外，API 应遵循统一响应结构原则。统一结构至少应包含以下语义层：

- 业务结果标识
- 人类可读消息
- 实际数据载荷
- 请求追踪标识
- 分页信息（列表接口）

设计意图如下：

- 前后端处理逻辑统一。
- 错误呈现方式统一。
- 便于日志与问题追踪。
- 后续接入网关或监控时结构稳定。

### 12.3 错误码体系建议

建议采用“HTTP 状态码 + 业务错误码”双层体系。

#### HTTP 状态码用途

- 成功请求返回成功状态。
- 参数错误、未授权、未找到、冲突、服务器异常分别映射标准状态码。
- 不将所有错误都包装成成功状态。

#### 业务错误码用途

业务错误码用于表达更细的系统语义，建议按域划分，例如：

- 公共查询类
- 后台鉴权类
- 采集执行类
- 资源处理类
- 系统内部类

#### 设计要求

- 错误码稳定，不因文案修改而变化。
- 同一类错误在公开端和后台端表现一致。
- 前端可根据错误码作精确交互，不依赖字符串判断。

### 12.4 初期不强制版本号的原因

一期不强制在 URL 层引入版本号，原因如下：

- 客户端只有公开前端和管理后台，均由同一项目控制。
- API 总量有限，演进成本可控。
- 过早引入版本号会增加接口路径和文档复杂度。

但不加版本号不代表忽略演进，必须做到：

- 响应字段新增尽量保持兼容。
- 避免随意修改既有字段语义。
- 需要破坏性变更时再引入版本路径或版本头。
- 接口文档和变更日志必须明确记录变更影响。

### 12.5 查询与筛选原则

公开接口查询应支持以下能力：

- 按日期排序
- 分页
- 按地区筛选
- 按分辨率范围筛选
- 按内容状态过滤时仅后台可用
- 默认只返回公开可见内容

后台接口查询应额外支持：

- 按任务状态
- 按处理状态
- 按失败类型
- 按来源
- 按创建时间区间

### 12.6 幂等性原则

- 自动采集和手动采集任务应支持重复触发但不产生重复内容。
- 状态切换接口应尽量幂等。
- 失败重试应有明确对象，不执行模糊批量重试。

## 13. 公开前端设计

### 13.1 目标

公开前端的目标是低门槛展示内容，不引入登录，不承载管理逻辑。其核心价值是“让用户快速找到并下载需要的壁纸”。

### 13.2 页面组成

#### 首页

用于展示最新壁纸和默认筛选入口。

应包含：

- 最新壁纸列表
- 简明筛选项
- 分页或加载更多
- 站点说明

#### 列表页

用于进行明确筛选和浏览。

应支持：

- 日期排序
- 地区筛选
- 分辨率筛选
- 来源筛选预留
- 空结果提示
- 分页导航

#### 详情页

用于展示单张壁纸完整信息。

应包含：

- 大图预览
- 标题与说明
- 版权信息
- 日期
- 地区
- 原始来源信息
- 下载入口
- 相关推荐预留

#### 错误页与空状态

需要明确设计以下状态：

- 数据为空
- 资源暂不可用
- 内容不存在
- 服务繁忙

### 13.3 前端边界

公开前端只做展示层，不承担以下责任：

- 不保存业务状态。
- 不绕过 API 直接查询数据库。
- 不处理后台权限。
- 不承担采集与资源恢复职责。

### 13.4 公开前端性能重点

- 列表页使用分页而非一次性加载全部。
- 缩略图与原图分离，避免列表页直接加载大图。
- 静态资源启用浏览器缓存。
- 详情页加载顺序优先保证核心信息，再加载非关键扩展信息。

## 14. 管理后台设计

### 14.1 目标

管理后台是系统运营入口，不追求复杂权限体系，但必须保证：

- 登录安全
- 状态控制清晰
- 操作可追踪
- 失败可定位

### 14.2 核心页面

#### 登录页

提供管理员认证入口。

#### 内容管理页

应支持：

- 列表查看
- 搜索与筛选
- 查看当前内容状态
- 查看资源处理状态
- 启用、禁用、删除
- 跳转查看详情

#### 内容详情页

应展示：

- 展示字段
- 来源字段
- 资源信息
- 当前状态
- 失败原因
- 最近操作记录

#### 采集任务页

应支持：

- 查看自动采集和手动采集记录
- 查看成功数、重复数、失败数
- 查看错误摘要
- 手动发起采集
- 对失败任务执行重试

#### 日志查看页

应支持：

- 按时间查询
- 按任务 ID 查询
- 按错误类型查询
- 快速查看最近失败事件

### 14.3 后台操作原则

- 危险操作必须二次确认。
- 删除采用逻辑删除，不直接物理删除文件。
- 所有管理动作写入审计日志。
- 手动采集必须记录发起人和参数快照。
- 后台不直接暴露底层文件系统路径给管理员下载浏览。

## 15. 定时任务与调度设计

### 15.1 调度职责

cron 仅负责“按时间触发”，不承担业务逻辑。  
具体采集逻辑在采集系统内执行。

### 15.2 任务类型

一期建议至少区分以下任务：

| 任务类型 | 说明 |
|---|---|
| 自动采集任务 | 按固定周期触发 |
| 手动采集消费任务 | 扫描并执行后台提交的待执行任务 |
| 失败重试任务 | 对符合条件的失败资源或失败任务执行重试 |
| 资源巡检任务 | 检查数据库记录与文件实际存在性是否一致 |
| 备份任务 | 按周期备份数据库和图片目录 |

### 15.3 调度原则

- 同一类采集任务应加互斥保护，避免并发重复执行。
- 任务执行结果必须落库。
- 单次任务异常退出时必须有清晰退出码和错误摘要。
- 失败重试应有次数上限，避免无限重试。
- 手动任务与自动任务复用同一业务流程。

### 15.4 为什么不引入消息队列

一期不引入消息队列，原因如下：

- 任务量小。
- 触发方式简单。
- cron 足以满足定时消费。
- 引入队列会显著增加部署和运维成本。

这是明确接受的技术债：后台手动采集不是严格实时，而是近实时执行。

## 16. 异常处理策略

异常处理必须做到“有分类、有落点、有恢复路径”。

| 异常类型 | 可能原因 | 处理策略 | 用户/管理员可见结果 |
|---|---|---|---|
| 网络失败 | 上游不可达、DNS、超时 | 标记任务失败，记录重试信息，不污染现有数据 | 后台可见失败原因 |
| 上游返回空数据 | 当日无数据、接口异常 | 记录为空结果任务，区分正常空数据与异常空数据 | 后台可见任务结果 |
| 上游格式变更 | 字段缺失、结构变化 | 中止解析，保存原始摘要，触发告警 | 后台显示“上游格式异常” |
| 图片下载失败 | 403、404、超时、文件损坏 | 资源状态置为 `failed`，内容保持非公开 | 前端不可见，后台可重试 |
| 数据重复 | 已存在相同日期地区或相同 URL | 记录为重复跳过，不视为系统错误 | 后台任务统计显示重复数量 |
| 文件写入失败 | 权限不足、磁盘满、路径错误 | 停止该条资源处理，记录失败原因 | 后台可见“文件写入失败” |
| 数据库写入失败 | 数据冲突、锁冲突、磁盘异常 | 回滚当前记录写入，保留任务失败信息 | 管理员可见失败详情 |
| 手动采集失败 | 参数错误、业务校验失败 | 拒绝创建任务或创建失败记录 | 后台即时提示 |
| 资源丢失 | 文件被删除、路径失效 | 巡检发现后标记资源异常，内容下线 | 前端不再显示该内容 |

### 处理原则

- 不可识别错误必须保留原始异常摘要。
- 任务级错误和条目级错误必须分开记录。
- 公开前端不暴露内部异常细节，只给出友好提示。
- 后台必须能看到足够的问题定位信息。

## 17. 日志、监控与健康检查

### 17.1 日志分类

日志不能只保留一类应用输出，建议至少区分以下类型：

| 日志类型 | 主要内容 |
|---|---|
| 访问日志 | API 请求路径、状态码、耗时、来源 IP、请求 ID |
| 应用日志 | 业务执行过程、状态流转、关键分支 |
| 采集日志 | 上游请求、去重结果、下载结果、任务汇总 |
| 错误日志 | 异常堆栈、失败原因、上下文信息 |
| 审计日志 | 管理员登录、状态变更、手动采集、删除操作 |
| 运维日志 | 备份、恢复、部署、巡检结果 |

#### 日志字段要求

所有日志至少包含：

- UTC 时间戳
- 日志级别
- 追踪 ID
- 模块名
- 事件类型
- 摘要信息

日志中不得输出密码、令牌、会话密钥等敏感信息。

### 17.2 指标建议

一期即使不引入完整监控平台，也应明确要采集的核心指标：

#### 采集类指标

- 每日采集成功数
- 重复数
- 失败数
- 平均耗时
- 图片下载失败率
- 上游空结果次数

#### API 类指标

- 请求总量
- 按接口维度的平均响应时间
- 4xx 比例
- 5xx 比例
- 公开列表接口耗时
- 后台登录失败次数

#### 存储类指标

- 数据库文件大小
- 图片目录容量
- 近七日增长量
- 磁盘剩余空间

### 17.3 健康检查

应至少提供三类健康检查：

| 类型 | 目的 | 检查内容 |
|---|---|---|
| 存活检查 | 判断服务进程是否运行 | 进程可响应 |
| 就绪检查 | 判断服务是否可提供业务 | 数据库可连接、关键目录可访问 |
| 深度检查 | 判断关键依赖是否正常 | 最近一次采集状态、磁盘空间、资源目录权限 |

#### 健康检查方法

- API 层提供健康检查接口。
- systemd 负责进程级巡检。
- 运维脚本负责备份与磁盘检查。
- cron 失败可通过退出状态和日志告警。

### 17.4 告警思路

一期可采用轻量告警机制，例如邮件、Webhook 或运维消息渠道。

建议告警条件：

- 连续多次自动采集失败
- 上游格式异常
- 图片下载失败率超过阈值
- 数据库不可写
- 磁盘空间低于阈值
- API 5xx 比例异常上升
- 最近备份失败

一期建议默认阈值：

- 连续 `3` 次自动采集失败
- 最近 `50` 次图片下载中失败率超过 `20%`
- 磁盘使用率超过 `85%`
- 最近 `5` 分钟内 API `5xx` 比例超过 `5%`
- 超过 `24` 小时未产生成功备份

## 18. 配置管理

### 18.1 配置原则

- 环境相关配置必须外置，不写死在代码中。
- 配置应区分开发、测试、生产环境。
- 敏感信息使用环境变量或受控配置文件注入。
- 配置项必须有默认值策略和校验规则。

### 18.2 关键配置项

建议管理以下配置类别：

| 类别 | 说明 |
|---|---|
| 服务配置 | 监听端口、基础 URL、时区显示策略 |
| 数据库配置 | 数据库文件路径、连接参数 |
| 存储配置 | 图片根目录、临时目录、备份目录 |
| 采集配置 | 来源开关、采集地区、超时、重试次数 |
| 安全配置 | 管理员会话密钥、密码策略、登录过期时间 |
| 日志配置 | 日志级别、日志目录、保留天数 |
| 告警配置 | 通知地址、阈值配置 |

一期安全配置基线说明：

- 会话密钥最少 `32` 字节随机值，使用 Base64 或十六进制受控存放
- 当前实现对初始化管理员密码仅强制“最少 `12` 位”；更严格的大小写 / 数字 / 特殊字符复杂度要求仍属于后续增强建议
- 当前密码摘要算法已固定为 `pbkdf2_sha256`；若后续升级到 `argon2id`，应作为显式安全演进处理，具体迁移边界见 [docs/password-hash-migration.md](/home/ops/Projects/BingWall/docs/password-hash-migration.md)
- 后台登录会话默认有效期建议不超过 `12` 小时

### 18.3 配置变更要求

- 配置变更必须可审计。
- 配置格式错误应在启动阶段即失败，不应运行时才暴露。
- 所有时间相关配置内部统一按 UTC 处理，展示层再本地化。

## 19. 缓存与性能策略

### 19.1 性能目标

一期性能目标不是追求极限吞吐，而是保证：

- 公开列表接口稳定响应。
- 图片下载链路轻量。
- 采集任务对在线查询影响可控。

### 19.2 缓存策略

#### 静态资源缓存

- 图片文件由 Nginx 提供缓存头。
- 文件名稳定时可使用较长缓存时间。
- 如未来存在资源替换，应在文件命名或路径层面体现版本差异。

#### API 查询缓存

一期不引入 Redis，优先采用以下策略：

- 数据库索引优化
- 合理分页
- 公开只读查询尽量简单
- 必要时在应用内对少量热点查询做短 TTL 缓存

#### 浏览器缓存

- 列表页接口可配置短时缓存语义。
- 静态前端资源启用浏览器缓存和压缩。

### 19.3 数据库性能策略

- 对常用筛选字段建立索引，例如日期、地区、内容状态、图片状态。
- 列表查询避免返回过多大字段。
- 统计查询与列表查询尽量分离。
- 后台日志查看应分页，避免一次读取大量日志。

### 19.4 明确不做的性能优化

一期不做以下内容：

- 分布式缓存
- 分片数据库
- 专用搜索引擎
- 图片实时转码集群

这些优化不符合一期规模，也会显著增加复杂度。

## 20. 安全与合规

### 20.1 安全目标

虽然系统没有公开用户体系，但仍存在后台登录、文件访问、接口暴露、上游输入不可信等风险，因此必须进行基础安全控制。

### 20.2 核心安全要求

#### 管理后台安全

- 管理员必须登录后才能访问后台接口。
- 密码必须以安全摘要形式保存，不保存明文。
- 登录失败应有限流或锁定策略。
- 会话必须有过期时间。
- 后台接口必须记录审计日志。

#### 输入校验

- 所有查询参数必须校验类型、长度、范围。
- 手动采集参数必须限制允许的来源、地区和日期格式。
- 上游返回的所有字段都视为不可信输入，必须做结构校验。
- 文件路径必须由系统生成，不接受客户端直接指定。

#### 文件安全

- 上传场景一期不存在，但下载文件仍需限制访问目录。
- Nginx 只暴露正式资源目录，不暴露临时目录和配置目录。
- 禁止通过路径拼接访问任意磁盘位置。

#### 接口安全

- 后台接口必须鉴权。
- 敏感接口可加 CSRF 或等效防护策略，具体取决于后台登录态实现。
- 应在 Nginx 或应用层设置基础限流，至少保护后台登录和高频查询接口。

### 20.3 合规边界

本项目主要处理公开图片及其元数据，重点关注：

- 保留来源信息和版权说明，不误导用户图片归属。
- 不伪造原创信息。
- 后台日志不记录敏感凭证。
- 下载链路不暴露服务器内部路径。

## 21. 存储、备份与恢复

### 21.1 备份对象

一期必须同时备份以下内容：

- SQLite 数据库文件
- 图片正式资源目录
- 关键配置文件
- 应用日志和审计日志
- 部署相关 service 与 Nginx 配置

### 21.2 备份策略

建议采用如下策略：

| 对象 | 频率 | 方式 |
|---|---|---|
| 数据库 | 每日 | 一致性备份 |
| 图片目录 | 每日增量，周期性全量 | 文件级备份 |
| 配置文件 | 变更后立即备份 | 版本留存 |
| 日志 | 每日归档 | 压缩保存 |

#### 关键要求

- SQLite 备份必须使用一致性方式，不直接在高写入过程中粗暴拷贝活跃文件。
- 图片和数据库备份应具有同一时间点语义，至少保证恢复时能够对应。
- 备份文件必须保留最近若干周期，避免单份损坏导致不可恢复。

### 21.3 恢复策略

恢复顺序建议如下：

1. 恢复配置文件。
2. 恢复数据库。
3. 恢复图片资源目录。
4. 启动应用。
5. 执行资源巡检，修复数据库与文件状态差异。
6. 验证公开查询与后台管理功能。

### 21.4 回滚原则

- 管理操作以逻辑删除和状态切换为主，便于快速回滚。
- 数据结构变更必须通过版本化迁移控制。
- 资源删除采用延迟清理机制，避免误删后无法恢复。

## 22. 数据迁移与模式演进

### 22.1 演进目标

系统必须允许从 SQLite 平滑演进到 MySQL 或 PostgreSQL，并支持从本地文件迁移到 OSS。

### 22.2 数据库演进原则

- 使用稳定的主键策略，不依赖 SQLite 特有行为。
- 时间统一用 UTC 存储。
- 避免过早使用仅某一数据库支持的高级特性。
- 所有模式变更必须通过版本化迁移管理，不手工改生产库。
- 破坏性变更必须提供迁移前验证与回滚说明。

### 22.3 从 SQLite 迁移到 MySQL 或 PostgreSQL 的关注点

- 字段类型兼容
- 索引语义一致
- 主键和唯一约束一致
- 文本排序与大小写规则差异
- 分页行为一致性
- 时间字段时区处理一致性

### 22.4 从本地文件迁移到 OSS 的关注点

- 资源表必须有存储后端字段。
- 资源访问地址不应直接等于磁盘路径。
- 应通过统一资源定位字段适配本地与 OSS。
- 迁移期间允许新旧资源共存，不要求一次性切换。

## 24. 部署方案

### 24.1 部署拓扑

一期采用单机部署，组件分布如下：

| 组件 | 部署方式 |
|---|---|
| Nginx | 系统服务 |
| FastAPI 应用 | systemd 托管进程 |
| SQLite | 本机文件 |
| 图片资源目录 | 本机磁盘 |
| cron | 系统定时任务 |

### 24.2 请求路径

- 公开前端静态资源由 Nginx 提供。
- 图片资源由 Nginx 提供静态访问。
- API 请求由 Nginx 反向代理到 FastAPI。
- 后台页面与后台 API 共用同一域名下不同路径或子路径。

### 24.3 部署要求

- 应用进程必须由 systemd 守护，支持自动重启。
- Nginx 必须区分静态文件、图片文件和 API 路由。
- 图片目录权限必须满足应用写入、Nginx 读取。
- 日志目录、备份目录、临时目录必须提前创建并赋权。
- 环境变量必须由受控方式注入。

### 24.4 上线检查清单

- 配置文件已核对
- 数据目录存在且权限正确
- 数据库可初始化
- 健康检查通过
- cron 已加载
- 首次备份已可执行
- 首次手动采集已验证

- 资源巡检
- 更细的下载统计
- OSS 适配
- 更完善的搜索能力

验收结果：

- 系统具备扩展基础，而不破坏一期主链路。

## 26. 风险与应对

| 风险 | 影响 | 应对方案 |
|---|---|---|
| Bing 上游接口变化 | 采集中断 | 解析层与业务层解耦，保留原始摘要并告警 |
| SQLite 锁冲突 | 写入失败 | 限制并发写入，任务串行化，控制后台批量操作 |
| 本地磁盘不足 | 下载失败、服务异常 | 监控磁盘空间，定期清理临时文件，启用备份归档 |
| 图片目录损坏或误删 | 公开资源不可用 | 资源巡检、延迟删除、定期备份 |
| 手动任务堆积 | 管理后台体验下降 | cron 分钟级消费，任务状态清晰，必要时增加执行频率 |
| 日志过大 | 磁盘膨胀 | 日志轮转与归档 |
| 后台账号泄露 | 管理风险 | 密码摘要、会话过期、登录限流、审计日志 |
| 单机故障 | 全站不可用 | 明确保留备份，恢复流程可演练，后续再扩展高可用 |

### 当前明确接受的技术债

- 单机单点故障
- SQLite 写并发能力有限
- 本地文件不可跨节点共享
- cron 缺乏复杂任务编排
- 后台手动采集为近实时而非即时执行

这些技术债不会阻止一期落地，但必须通过日志、备份、巡检和明确的升级路径进行管理。

## 27. 后续扩展方向

系统后续可沿以下方向扩展：

### 多来源采集

在 Bing 之外增加更多图片来源，要求采集层和去重层已按“来源类型”抽象。

### 标签与搜索

为内容增加标签体系、关键词索引和更丰富的筛选能力。

### OSS 与 CDN

将本地文件迁移到对象存储，并通过 CDN 提升访问性能和稳定性。

### 下载统计与内容分析

统计下载次数、热门内容、地区偏好、采集成功率趋势等指标。

### 图片衍生资源

增加缩略图、移动端尺寸、WebP 等派生版本。

### 更完善的任务系统

当采集来源增多、任务复杂度上升后，再引入专用任务队列和执行器。

## 28. 项目总结

BingWall 一期的核心不是“抓一张图片”，而是建立一套可以持续运行的内容系统基础设施。该系统以单机架构为前提，围绕采集、存储、接口、展示、管理五个部分构建完整闭环，并通过清晰的状态模型、去重策略、资源策略和运维保障，使系统在当前规模下具备可实施性和可维护性。

本方案的关键判断如下：

- SQLite 在当前阶段可行，因为业务是低频写入、轻量内容管理、单机部署。
- 本地文件存储在当前阶段可行，因为资源规模可控，Nginx 可高效提供静态访问。
- cron 在当前阶段可行，因为任务种类少、调度简单、无需复杂编排。
- 单机架构在当前阶段合理，因为目标是优先完成稳定闭环，而不是过早追求复杂扩展性。

在此基础上，系统已为后续扩展到多来源、OSS、搜索、标签和统计能力预留了清晰路径。该设计可以直接作为后续开发、接口实现、页面实现、部署实施和项目管理的依据。
